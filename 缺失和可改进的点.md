## 🔴 比较严重缺失

### 1. 协议实现安全机制

17 种协议（16 个协议目录，Modbus 含 TCP/RTU 两种变体）在 README 里风风火火，但没有提到：

| 协议 | 安全现状 | 说明 |
|------|---------|------|
| Modbus/TCP | ❌ 无身份验证 | 任何人都能读写寄存器，这是 Modbus 协议本身的设计限制 |
| OPC-UA | ⚠️ 依赖配置 | asyncua 支持证书和加密通道，但模板默认未启用 |
| MQTT | ⚠️ 依赖配置 | aMQTT 支持用户名/密码认证，但模板默认未启用 |
| GB28181 | ⚠️ 部分支持 | 支持 Digest 认证，但 RTP 流未加密（SRTP 未实现） |
| S7/MC/FINS/AB | ❌ 无身份验证 | 工业协议通常在 PLC 端做访问控制 |
| HTTP | ✅ 有认证 | 使用 JWT + RBAC 保护 API |
| BACnet | ❌ 无身份验证 | BACnet/IP 协议本身无内置认证 |

**建议**：
- 在文档中明确各协议的安全边界
- OPC-UA 模板增加安全模式选项（None/Sign/SignAndEncrypt）
- MQTT Broker 增加认证配置项
- GB28181 增加 SRTP 加密传输选项
- 录制回放功能：录制的报文可能含敏感信息，存储时应考虑加密

### 2. Webhook 通知和告警规则系统——已实现但完全未文档化

代码中实现了完整的 Webhook + 告警反应规则体系，但缺失文档和 README 均未提及：

- ✅ 事件订阅（按事件类型过滤，支持通配符 `*`）
- ✅ 异步发送队列（5000 条消息缓冲，异步批量发送）
- ✅ HMAC-SHA256 签名验证（`X-ProtoForge-Signature` 头）
- ✅ 统计信息（触发次数、错误次数、队列大小）
- ✅ 测试端点（`POST /webhooks/{id}/test`）

**告警反应规则**（`protoforge/core/integration/manager.py`）：

- ✅ 自动反应动作：`stop_device`（停止设备）/ `inject_fault`（注入故障）/ `adjust_generator`（调整生成器）
- ✅ EdgeLite 告警转发（`alarm_fired` / `alarm_recovered`）
- ✅ API 端点：`GET/POST /api/v1/integration/alarm-rules`、`DELETE /api/v1/integration/alarm-rules/{id}`

**场景规则触发 Webhook**（`protoforge/core/scenario.py`）：
- ✅ 场景规则（阈值/值变化/定时/脚本）触发时自动通知 Webhook

**建议**：
- 在 README 和缺失文档中补充 Webhook 和告警规则的完整说明
- 提供 Webhook 配置和使用示例
- 补充告警规则与 EdgeLite 的联动说明

### 3. 无集群/高可用方案

目前是单实例架构：一个后端 + 一个数据库

**生产环境常见问题**：
- 后端挂了怎么办？没有故障切换
- WebSocket 状态是内存存储，重启就丢失
- 数据库主从复制配置缺失
- Docker Compose 是单机的，没有 K8s/Helm 支持

**已实现的启动时自动恢复**（`protoforge/main.py`）：
- ✅ 设备恢复：从数据库加载所有设备并重新创建到引擎中
- ✅ 场景恢复：从数据库加载场景配置并重建
- ✅ 模板恢复：从数据库加载自定义模板
- ✅ 测试数据恢复：恢复测试用例、测试套件、测试报告
- ✅ 用户恢复：从数据库恢复用户账号（保留 admin 默认用户）
- ✅ 恢复错误收集：所有恢复错误被收集，启动后以 warning 日志汇总报告

**仍缺失**：
- ❌ 数据库备份/导出 API 或定时备份功能
- ❌ 故障切换机制

**建议**：
- 增加 Redis 作为 WebSocket 状态后端（支持多实例共享）
- 提供 Helm Chart 用于 K8s 部署
- PostgreSQL 配置主从复制文档
- 考虑使用 Gunicorn + 多 worker 部署

### 3. 性能基准和资源消耗未提及

一台机器最多能跑多少仿真设备？

| 场景 | 预估参考值 | 说明 |
|------|-----------|------|
| Modbus TCP 并发连接 | ~200 连接 | 受限于文件描述符和内存 |
| MQTT 并发客户端 | ~500 连接 | aMQTT 性能有限，生产建议用 Mosquitto |
| OPC-UA 并发会话 | ~50 连接 | asyncua 资源消耗较大 |
| GB28181 RTP 推流 | ~10 路 | 受 CPU 和带宽限制 |
| 内存占用（空载） | ~150MB | Python + FastAPI 基础开销 |
| 内存占用（100 设备） | ~500MB | 含协议栈和仿真数据 |
| CPU 占用（空载） | <5% | 等待连接状态 |
| CPU 占用（100 设备活跃） | 30-60% | 取决于数据更新频率 |

> ⚠️ 以上为估算值，实际性能取决于硬件配置、协议类型和数据更新频率。建议在目标环境进行基准测试。

## 🟡 一般缺失

### 4. 监控和可观测性

虽然提到了 Prometheus metrics，但缺少：

- **Grafana Dashboard 模板** — 用户如何快速搭建监控看板？

- **告警规则示例** — 什么情况该告警？
- **链路追踪** — 一次完整的 GB28181 调用链如何追踪？

**当前已实现的监控端点**：
- `GET /health` — 健康检查（数据库状态、活跃设备数、协议状态）
- `GET /metrics` — Prometheus 格式指标（uptime、设备数、转发计数、录制计数）
- `GET /api/v1/forward/stats` — 转发统计
- `GET /api/v1/recorder/stats` — 录制统计

**建议补充**：
- 提供 Grafana Dashboard JSON 模板
- 提供告警规则 YAML 示例
- 增加请求链路追踪（OpenTelemetry 集成）

### 6. 设备模板市场机制

53 个模板都是内置的，用户如何自定义模板并持久化？

**当前状态**：
- ✅ 模板以 JSON 文件存储在 `protoforge/templates/` 目录
- ✅ 用户可通过 API 创建自定义模板
- ✅ 模板支持标签分类和搜索
- ❌ 自定义模板不持久化（重启丢失，除非写入文件）

- ❌ 社区模板无法共享/导入导出

**建议**：
- 自定义模板持久化到数据库
- 增加模板导入/导出 API
- 增加模板版本管理
- 考虑社区模板市场

## 总结

经过八轮系统审计，ProtoForge 的核心功能链路已经非常完整：

- ✅ 17 种工业协议仿真
- ✅ 53 个设备模板
- ✅ JWT + RBAC 四角色认证（100% 端点覆盖）
- ✅ WebSocket 实时日志（Token 认证 + 心跳 + 登出关闭）
- ✅ 数据转发（InfluxDB/HTTP/File + 重试机制）
- ✅ Prometheus 监控指标
- ✅ 协议录制回放
- ✅ 双数据库支持（SQLite/PostgreSQL）
- ✅ Docker 一键部署
- ✅ Python SDK（70+ 方法，同步/异步双客户端）
- ✅ EdgeLite 网关对接
- ✅ CI/CD 流水线

主要短板是**安全机制的系统性描述**和**生产环境的性能数据**，这两块补齐后，用户会更放心地用到正式项目中。
